# Trusted Query Boundary

## 目标

这份文档用于说明：

1. 当前系统哪些查询行为是可信的
2. 哪些能力已经支持
3. 哪些自动化能力暂不开放

## 1. 当前可信能力

当前系统可信的能力包括：

1. 显式 `Query Intent` 查询
2. 基于图谱的受控字段关联
3. 标准字段到 SQL 的稳定映射
4. 返回 SQL 可审计
5. 返回 DataFrame 可直接消费

## 2. 当前可信前提

### 查询时需要明确：

1. `from`
2. `select`
3. `where`
4. `time_range`
5. `safety`

### 不依赖：

1. 自动猜底层表名
2. 自动自由拼接 SQL
3. 无边界跨图谱乱关联

## 3. 当前已支持的约束

### 路径约束

1. 多路径字段通过 `path_domain/path_group` 控制
2. 冲突路径组合会拒绝，不会硬查

### 时间语义

1. 日期类字段已补充 `time_semantics`
2. 不同公告日/交易日语义可以区分

### lookahead 第一阶段

当前 `lookahead_safe = true` 仅对：

1. 公告类字段
2. 事件类字段
3. 条款生效类字段

生效。

当前明确不影响：

1. `close`
2. `high`
3. `low`
4. `volume`
5. `amount`
6. `market_cap`
7. `turnover_rate`

## 4. 当前不承诺完全自动化的部分

以下能力当前不作为正式可信能力开放：

1. 全自动字段驱动选入口
2. 全自动跨多个业务入口合并查询
3. 完全自动拆查询
4. 自动猜测分钟/日线粒度

## 5. 当前系统的可信边界

一句话定义：

**当前系统在“主入口明确 + 标准字段明确 + 图谱路径明确”的前提下是可信的。**

如果超出这个边界，系统不应该硬查，而应该：

1. 报冲突
2. 报不支持
3. 建议拆查询

## 6. 推荐使用方式

1. 通过主入口节点发起查询
2. 通过标准字段选择查询内容
3. 保留 SQL 输出用于审计
4. 避免把多个业务主体强行塞进单条查询
