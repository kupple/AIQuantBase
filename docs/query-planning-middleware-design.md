# 关系图驱动查询规划中间件设计 Demo

> 状态：未敲定，仅作为当前阶段的设计草案与讨论样例，不视为最终定稿。

## 1. 文档目标

本文档是一个 `demo` 级别的设计说明，用来帮助讨论系统方向、模块边界和配置思路。

当前版本的目的不是直接冻结协议，而是先把核心骨架表达清楚，便于后续继续调整。

本项目的核心定位也需要明确记录下来：

**本项目用于将 A 股研究数据库中复杂的表关系、时间语义和研究约束进行统一规划，形成可由 AI 稳定调用的查询协议与关系图驱动查询中间层。**

换句话说，这个项目的价值不只是“查数据”，而是：

**把数据库中复杂、分散、时序不统一的关系整理成可规划、可复用、可安全执行的研究查询体系。**

本文档定义的新系统，不是传统的 SQL 编辑器，也不是单张标准宽表系统。

它的目标是构建一个：

**基于关系图、时间语义和配置驱动的高速查询中间件。**

这个中间件位于“上层研究表达”和“底层数据库表”之间，负责：

1. 接收查询请求
2. 识别查询入口表
3. 根据配置自动寻找关联路径
4. 自动处理实体映射、时间对齐和防未来函数
5. 自动生成正确、稳定、可优化的 SQL

---

## 2. 要解决的问题

当前系统面临的问题，不只是股票日线关联 `ST`、后复权、股本等数据。

更普遍的问题是：

1. 查询入口不止一张表
2. 不同表之间的关系类型不同
3. 时间字段语义不统一
4. 很多表不是一对一关系，而是一对多或多对多
5. 财务、公告、新闻等表容易产生未来函数问题

因此，系统不能只围绕：

1. `stock_daily`
2. `stock_minute`
3. 标准字段宽表

来设计。

更适合的方式是：

**把数据库中的表组织成一张“带时间语义的关系图”，查询时自动在图中找路径并生成 SQL。**

---

## 3. 核心定位

这个系统的本质是：

**查询规划中间件**

而不是：

1. 自然语言转 SQL 工具
2. 静态标准字段字典
3. 单一研究宽表生成器

宽表未来可以作为某些高频查询的物化结果，但不是系统核心。

系统核心应该是：

1. `node`
2. `edge`
3. `query plan`

---

## 4. 核心设计思想

### 4.1 数据库不是表集合，而是关系图

每一张可查询表，都应被抽象成一个 `node`。

例如：

1. 股票日线
2. 股票分钟线
3. 指数日线
4. 行业指数日线
5. ST 状态表
6. 股本快照表
7. 财务表
8. 新闻表
9. 行业映射表
10. 指数成分表

每一条可配置的关联关系，都应被抽象成一条 `edge`。

例如：

1. 股票日线 -> ST 状态
2. 股票日线 -> 股本快照
3. 股票分钟线 -> 财务表
4. 指数日线 -> 指数新闻
5. 行业指数 -> 行业映射 -> 行业属性

查询时，中间件不是直接拼 SQL，而是：

1. 找到入口节点
2. 找到目标字段所属节点
3. 在图中寻找可用路径
4. 按路径生成 join 和时间过滤
5. 输出最终 SQL

### 4.2 配置描述“关系”，不是描述“SQL”

配置文件不应该要求直接写复杂 SQL。

更好的方式是通过结构化配置描述：

1. 两张表如何连
2. 用哪些键连
3. 时间如何对齐
4. 数据何时生效
5. 是否允许未来数据

SQL 生成应由规划器和生成器负责。

### 4.3 入口表可以有很多

系统不能把查询入口固定成股票日线。

查询入口可以是：

1. 股票日线
2. 股票分钟线
3. 指数日线
4. 行业指数日线
5. 新闻表
6. 公告表
7. 事件表

因此系统必须支持：

**多入口节点**

### 4.4 对齐优先于字段标准化

第一阶段优先解决：

**表和表如何可靠对齐**

而不是先做：

1. 大而全的标准字段体系
2. 全量派生公式系统
3. 全市场统一宽表

只要关系图和时间语义稳定，后续字段暴露、宽表沉淀、缓存优化都会变容易。

## 5. 三个核心对象

### 5.1 Node

`node` 表示一个可查询的数据节点。

它通常对应：

1. 一张物理表
2. 一张逻辑表
3. 一个预处理后的物化表

`node` 的作用是告诉系统：

1. 这是什么实体
2. 它的主键是什么
3. 它的时间键是什么
4. 它属于哪种时间粒度

典型节点示例：

1. `stock_daily`
2. `stock_minute`
3. `index_daily`
4. `industry_index_daily`
5. `stock_st_status`
6. `stock_capital_snapshot`
7. `stock_revenue_report`
8. `index_news`

### 5.2 Edge

`edge` 表示两个节点之间的一条可用关系。

它的作用是告诉系统：

1. 起点节点是谁
2. 终点节点是谁
3. 通过哪些键建立映射
4. 时间如何对齐
5. 是否需要桥接表
6. 是否允许未来数据

一条边不是 SQL，而是：

**可规划、可校验、可复用的关系语义。**

### 5.3 Query Plan

`query plan` 表示一次查询如何从入口节点出发，到达目标节点并取回结果。

它至少包含：

1. 入口节点
2. 目标字段
3. 所需路径
4. 每一跳的 join 策略
5. 时间对齐策略
6. 未来函数约束

---

## 6. 关系类型

第一版建议只支持两种大的关系类型。

### 6.1 Direct

直接关系。

含义：

两张表可以直接建立关联，不需要第三张桥接表。

适用场景：

1. 股票日线 -> ST 状态
2. 股票日线 -> 股本快照
3. 指数日线 -> 指数新闻
4. 股票分钟线 -> 股票日频标签

### 6.2 Bridge

桥接关系。

含义：

起点表和终点表无法直接对应，需要经过一张中间映射表。

适用场景：

1. 行业指数 -> 行业映射 -> 行业属性
2. 指数 -> 成分股映射 -> 股票属性
3. 股票 -> 概念映射 -> 概念标签
4. 股票 -> 行业成分映射 -> 行业表

桥接关系是支持一对多、多对一、多对多的关键。

---

## 7. 时间对齐语义

时间对齐是整个中间件的重点。

第一版建议支持以下几类。

### 7.1 same_date

含义：

同一实体、同一日期直接对应。

适用：

1. 日频状态表
2. 日频标签表
3. 日频新闻聚合表

### 7.2 same_timestamp

含义：

同一实体、同一时间戳直接对应。

适用：

1. 分钟表对分钟表
2. 秒级表对秒级表

### 7.3 asof

含义：

在同一实体下，取不晚于基准时间的最近一条记录。

适用：

1. 股本快照
2. 财务已披露值
3. 公告后生效指标
4. 非每日更新的属性表

这是处理“日期不齐取最近日期”的核心机制。

### 7.4 effective_range

含义：

基准时间落在某个生效区间内。

适用：

1. ST 区间表
2. 行业映射区间表
3. 指数成分区间表
4. 停牌状态区间表

---

## 8. 防未来函数

系统必须内建防未来函数能力。

这是配置层必须具备的约束，不应交给每次查询临时处理。

### 8.1 原则

任何被关联到基准节点的数据，都必须满足：

**在基准时间点，该数据已经对市场可见。**

### 8.2 财务类风险

例如营收表可能有：

1. `report_period`
2. `announce_date`
3. `revenue`

系统不能按 `report_period` 直接与交易日对齐，因为：

报告期结束时，市场并不知道财报内容。

正确方式应该是：

1. 以 `announce_date` 作为可见时间
2. 再用 `asof` 对齐到交易日或分钟时间

### 8.3 通用规则

对于任何存在“记录时间”和“市场可见时间”差异的表，应优先配置：

1. `available_time`
2. 或与之等价的“可见时间字段”

规划器在生成 SQL 时，应始终以“可见时间”而不是“业务归属时间”参与防未来函数判断。

---

## 9. 为什么不把所有逻辑都交给公式

对齐系统不应主要依赖自由公式。

原因：

1. 公式可读性差
2. AI 生成不容易统一
3. 很难做自动校验
4. 很难检测未来函数
5. 很难做查询规划和路径选择

因此推荐策略是：

1. 对齐关系优先使用固定关系类型
2. 时间语义优先使用结构化字段描述
3. 公式只作为少量特殊场景的兜底能力

也就是说：

1. “怎么连”优先结构化
2. “连完怎么算”才考虑公式

---

## 10. 查询规划流程

一个查询请求进入中间件后，建议按以下步骤执行：

1. 识别入口节点
2. 识别目标字段所在节点
3. 在关系图中寻找可达路径
4. 评估每一跳关系是否合法
5. 检查时间语义是否满足
6. 检查是否存在未来函数风险
7. 选择更优路径
8. 生成 SQL

如果未来支持性能优化，还可以在规划阶段加入：

1. 是否优先使用物化表
2. 是否优先使用预对齐表
3. 是否限制大表桥接路径

## 11. 第一阶段范围

第一阶段不追求完整中间件所有能力。

优先聚焦：

1. 定义 `node` 模型
2. 定义 `edge` 模型
3. 定义时间对齐类型
4. 定义防未来函数规则
5. 支持 `direct` 和 `bridge`
6. 让系统能从多入口节点自动规划基础查询路径

第一阶段先不追求：

1. 完整公式系统
2. 自动代价优化器
3. 全量物化策略
4. 任意复杂 DSL
5. 所有资产类别统一支持

## 12. 第一阶段示例场景

第一阶段应至少覆盖以下几种典型查询。

### 12.1 股票日线 -> ST 状态

目标：

从股票日线出发，关联当日或当时有效的 `is_st`。

关系特点：

1. `direct`
2. `same_date` 或 `effective_range`

### 12.2 股票日线 -> 股本快照

目标：

从股票日线出发，关联最近可用股本。

关系特点：

1. `direct`
2. `asof`

### 12.3 股票分钟线 -> 财务营收

目标：

从股票分钟线出发，查询当时已披露的最近营收值。

关系特点：

1. `direct`
2. `asof`
3. 基于 `announce_date` 防未来函数

### 12.4 行业指数日线 -> 行业属性

目标：

从行业指数日线出发，经映射表关联行业属性。

关系特点：

1. `bridge`
2. `effective_range`

### 12.5 指数日线 -> 当天新闻

目标：

从指数日线出发，关联当天新闻或新闻聚合结果。

关系特点：

1. `direct`
2. `same_date`

## 13. 架构结论

这个项目的正确方向，不是先做一张万能标准宽表，也不是让 AI 直接面向 SQL 工作。

更合理的系统骨架应该是：

1. 用 `node` 抽象所有可查询表
2. 用 `edge` 抽象所有合法关系
3. 用时间语义控制数据对齐
4. 用可见时间规则防未来函数
5. 用查询规划器自动拼接路径
6. 最终生成 SQL

一句话总结：

**要构建的是一个带时间语义、可规划路径、支持多入口和多对多关系的高速查询中间件。**

---

## 14. 最终完整链路

系统最终目标不是让 AI 直接生成 SQL，而是走一条分层链路：

1. 自然语言
2. `Query Intent`
3. `Query Plan`
4. SQL
5. 查询结果

其中：

### 14.1 Query Intent

`Query Intent` 是 AI 与中间件之间的标准查询协议。

它只描述：

1. 从哪个入口节点出发
2. 想要哪些字段
3. 过滤条件是什么
4. 是否分组、排序、限制行数
5. 是否要求严格防未来函数

它不描述：

1. join 哪些表
2. 如何走关系图
3. 时间如何对齐
4. SQL 如何写

### 14.2 Query Plan

`Query Plan` 是中间件内部的执行计划。

它描述：

1. 字段在哪些节点
2. 入口节点到目标节点的路径
3. 每一跳使用哪条边
4. 时间如何对齐
5. 如何防未来函数
6. SQL 最终应如何组织

因此应明确区分：

1. `Query Intent` 是协议层
2. `Query Plan` 是执行规划层

## 15. SQL 中间件设计

SQL 中间件应设计成一个分层规划器，而不是单一黑盒。

建议最少包含以下模块。

### 15.1 Schema Graph Registry

职责：

1. 管理 `node`
2. 管理 `edge`
3. 管理字段归属
4. 管理关系类型
5. 管理时间语义

### 15.2 Intent Validator

职责：

1. 校验 `Query Intent` 是否合法
2. 检查入口节点是否存在
3. 检查字段是否存在
4. 检查过滤条件是否合法
5. 检查是否违反未来函数规则

### 15.3 Path Planner

职责：

1. 从入口节点到目标节点寻找路径
2. 选择更优路径
3. 决定是否使用桥接关系
4. 决定每一跳的 join 方案

### 15.4 Temporal Resolver

职责：

1. 处理 `same_date`
2. 处理 `same_timestamp`
3. 处理 `asof`
4. 处理 `effective_range`
5. 应用 `available_time` 防未来函数

时间逻辑应独立，不应散落在 SQL 拼接代码中。

### 15.5 SQL Renderer

职责：

1. 把 `Query Plan` 渲染为 SQL
2. 生成 `FROM`
3. 生成 `JOIN`
4. 生成 `SELECT`
5. 生成 `WHERE / GROUP BY / ORDER BY`

`SQL Renderer` 不负责猜测关系路径，只负责翻译规划结果。

### 15.6 Execution Adapter

职责：

1. 执行 SQL
2. 管理分页
3. 管理超时
4. 管理缓存键
5. 返回结果

## 16. AI 辅助图谱构建 CLI

除了查询链路本身，系统还应提供一套 CLI，用于构建和维护关系图谱。

这套 CLI 的作用不是执行查询，而是：

**从数据库元信息出发，结合 AI 推断，生成可用的节点和关系配置。**

### 16.1 建图链路

建议单独存在一条“建图链路”：

1. CLI 读取白名单配置
2. 连接数据库
3. 拉取允许范围内的 schema 元信息
4. 结合 AI 生成候选节点和候选关系
5. 人工确认后沉淀为正式图谱配置

### 16.2 为什么需要 CLI

如果图谱完全手工维护，随着表越来越多，成本会迅速上升。

CLI 的价值是：

1. 自动采集真实 schema
2. 降低手工配置成本
3. 让 AI 基于真实数据库信息推断关系
4. 保持图谱和底层数据库同步

### 16.3 白名单机制

CLI 不应默认扫描全库。

它应严格受以下配置约束：

1. `allow_database`
2. `allow_table`

CLI 只处理允许范围内的库和表。

这样可以保证：

1. 安全
2. 可控
3. 降低噪音
4. 提高 AI 推断准确度

### 16.4 CLI 采集的元信息

CLI 至少应采集以下信息：

1. 数据库名
2. 表名
3. 字段名
4. 字段类型
5. 字段注释
6. 主键候选
7. 时间字段候选
8. 分区键或排序键
9. 行数规模估计

### 16.5 AI 推断的职责

AI 在建图阶段不直接生成 SQL。

它更适合做以下工作：

1. 判断一张表更像 `node` 还是 `bridge`
2. 判断实体类型
3. 判断主实体键候选
4. 判断时间键候选
5. 判断时间粒度
6. 判断可能的关系边
7. 判断是否存在未来函数风险

### 16.6 三层结果

建议把建图结果分成三层：

1. `Observed`
   数据库真实采集结果
2. `Inferred`
   AI 推断出的候选图谱
3. `Approved`
   人工确认后的正式图谱配置

这样可以避免 AI 推断结果未经审查直接进入生产。

### 16.7 CLI 命令建议

第一版 CLI 可以考虑提供以下能力：

1. `graph discover`
   拉取白名单范围内的 schema 信息
2. `graph infer-nodes`
   生成候选节点定义
3. `graph infer-edges`
   生成候选关系定义
4. `graph review`
   展示候选图谱
5. `graph export`
   导出正式图谱配置
6. `graph doctor`
   检查孤立节点、冲突边、未来函数风险和缺失时间键

### 16.8 CLI 与查询中间件的分工

两者职责必须分开：

1. CLI 负责建图
2. 查询中间件负责用图

也就是说：

1. CLI 是离线配置生成工具
2. Planner 是在线查询执行核心

---

## 17. 系统闭环

整个系统最终应形成一个完整闭环：

1. CLI 在白名单范围内连接数据库并采集 schema
2. AI 基于采集结果生成候选图谱
3. 人工确认后沉淀为正式 `node / edge` 配置
4. 用户使用自然语言提出查询需求
5. AI 将自然语言转换为 `Query Intent`
6. 查询中间件基于关系图生成 `Query Plan`
7. SQL 渲染器生成 SQL
8. 执行器获取结果并返回

这个闭环的关键收益是：

1. AI 不直接依赖 SQL 细节
2. 查询生成更稳定
3. 图谱可持续演进
4. 多入口、多对多和时间语义问题可统一处理

## 18. 功能开发范围 Demo

从功能开发角度看，本项目不是单一模块，而是一套围绕 A 股股票数据底座展开的能力组合。

第一版建议按以下功能模块拆分。

### 18.1 图谱配置管理

负责把数据库中的复杂表关系整理成可规划图谱。

主要功能：

1. 定义 `Node`
2. 定义 `Edge`
3. 管理字段归属
4. 管理实体键与时间键
5. 管理直接关系与桥接关系
6. 管理时间对齐规则

### 18.2 时间对齐与研究安全

负责处理研究查询中最关键的时序问题和未来函数问题。

主要功能：

1. 支持 `same_date`
2. 支持 `same_timestamp`
3. 支持 `asof`
4. 支持 `effective_range`
5. 基于 `available_time` 控制可见性
6. 对财务公告类数据做防未来函数检查

### 18.3 Query Intent 协议层

负责定义 AI 与中间件之间的标准查询协议。

主要功能：

1. 定义查询入口节点
2. 定义查询字段
3. 定义过滤条件
4. 定义时间范围
5. 定义分组、排序、限制条数
6. 定义查询安全约束

### 18.4 查询规划器

负责把 `Query Intent` 转换成内部执行计划。

主要功能：

1. 字段归属识别
2. 关系路径查找
3. 路径优先级选择
4. 时间对齐策略应用
5. 未来函数检查
6. 生成 `Query Plan`

### 18.5 SQL 生成与执行

负责把执行计划转换为 SQL 并获取结果。

主要功能：

1. 生成 `SELECT`
2. 生成 `FROM`
3. 生成 `JOIN`
4. 生成 `WHERE / GROUP BY / ORDER BY`
5. 执行 SQL
6. 返回结果
7. 预留分页、缓存、超时控制能力

### 18.6 CLI 图谱构建工具链

这是本项目必须具备的功能，不只是附属工具。

CLI 的职责是：

**连接数据库、读取白名单配置、采集 schema 元信息，并结合 AI 生成候选图谱配置。**

主要功能：

1. 根据 `allow_database` 和 `allow_table` 限定扫描范围
2. 连接数据库读取库表信息
3. 采集字段、类型、注释、键候选、时间列候选等元信息
4. 调用 AI 生成候选 `Node`
5. 调用 AI 生成候选 `Edge`
6. 输出候选配置供人工审核
7. 导出正式图谱配置
8. 提供图谱体检能力，如孤立节点、冲突关系、未来函数风险检查

可以预期的 CLI 子命令包括：

1. `graph discover`
2. `graph infer-nodes`
3. `graph infer-edges`
4. `graph review`
5. `graph export`
6. `graph doctor`

### 18.7 A 股研究语义增强

负责让本项目真正成为 A 股股票数据底座，而不是通用 join 中间件。

主要功能：

1. 支持股票、指数、行业、概念、公告、新闻等实体
2. 支持日线、分钟线、快照、区间、事件、财报等时间结构
3. 支持 ST、复权、股本、行业、财务公告等高频研究关系
4. 为后续 `panel_query`、`cross_section`、`event_study` 等研究模式提供基础支撑

### 18.8 建议开发顺序

建议第一版按以下顺序推进：

1. 图谱配置模型
2. 时间对齐与研究安全
3. `Query Intent` 协议
4. 查询规划器
5. SQL 生成与执行
6. CLI 图谱构建工具链
7. A 股研究语义增强与高频预制关系

## 19. Query Intent Demo

当前阶段建议优先把 `Query Intent` 定义为：

**AI 与查询中间件之间的最小查询协议。**

它的职责是表达“想查什么”，而不是表达“怎么查”。

### 19.1 Query Intent 的边界

`Query Intent` 负责描述：

1. 从哪个入口节点出发
2. 查询哪些字段
3. 过滤哪些条件
4. 时间范围是什么
5. 如何排序
6. 取多少条数据
7. 是否要求严格防未来函数

`Query Intent` 不负责描述：

1. join 哪些表
2. 走哪些关系边
3. 用什么时间对齐策略
4. SQL 语法细节

这些都属于中间件内部的 `Query Plan` 和 SQL 生成器职责。

### 19.2 第一版最小字段建议

第一版建议先只保留以下字段：

1. `from`
   入口节点
2. `select`
   目标字段列表
3. `time_range`
   查询时间范围
4. `where`
   普通过滤条件
5. `order_by`
   排序规则
6. `limit`
   返回行数
7. `safety`
   查询安全约束

### 19.3 Demo 结构示意

以下结构仅为 `demo`，用于讨论协议形态，不代表最终定稿：

```yaml
from: stock_daily

select:
  - close
  - is_st
  - revenue

time_range:
  field: trade_date
  start: "2024-01-01"
  end: "2024-03-31"

where:
  - field: code
    op: "="
    value: "600519.SH"

order_by:
  - field: trade_date
    direction: asc

limit: 1000

safety:
  lookahead_safe: true
  strict_mode: true
```

### 19.4 字段解释

#### `from`

表示查询从哪个入口节点开始。

例如：

1. `stock_daily`
2. `stock_minute`
3. `index_daily`
4. `news_event`

#### `select`

表示希望返回哪些字段。

这里引用的是系统对外暴露的研究字段名，而不是底层数据库原始字段名。

例如：

1. `close`
2. `is_st`
3. `revenue`
4. `industry_name`

#### `time_range`

表示查询的主时间范围。

建议单独抽出，而不是完全混在 `where` 里，因为时间范围是研究查询最核心的条件之一。

#### `where`

表示普通过滤条件。

例如：

1. 股票代码过滤
2. 指数代码过滤
3. 行业过滤
4. 状态过滤

#### `order_by`

表示结果如何排序。

第一版只建议支持简单字段排序。

#### `limit`

表示最多返回多少条记录。

#### `safety`

表示查询的安全要求。

第一版最重要的是：

1. `lookahead_safe`
2. `strict_mode`

其中：

1. `lookahead_safe = true` 表示必须防未来函数
2. `strict_mode = true` 表示规划器应优先采用研究上更安全的路径

### 19.5 为什么先做最小协议

第一版不建议一开始把 `Query Intent` 设计得过大。

原因：

1. AI 更容易稳定输出
2. 更容易做校验
3. 更容易和 `Query Plan` 解耦
4. 更容易在真实使用中逐步扩展

### 19.6 后续可扩展方向

等第一版稳定后，后续可以逐步增加：

1. `group_by`
2. `aggregations`
3. `window`
4. `event_anchor`
5. `universe`
6. `research_mode`

但这些都建议在第一版最小协议跑通后再考虑。

## 20. 图谱前端草案

当前项目非常适合补一个前端，但前端的第一阶段定位不应是“重编辑器”，而应是：

**图谱查看与审核台**

### 20.1 为什么要做前端

随着节点、边、标准字段和派生字段逐渐增多，单纯依赖 YAML 和 JSON 会出现几个问题：

1. 关系不直观
2. 候选图谱难审核
3. 孤立节点和错误关系不容易发现
4. AI 候选结果与人工确认结果不容易比较
5. 多人协作维护成本高

因此，前端的价值主要不是“替代配置文件”，而是：

**帮助人更高效地理解、审查和确认图谱。**

### 20.2 第一阶段定位

第一阶段建议只做：

1. 查看图谱
2. 查看节点详情
3. 查看边详情
4. 查看标准字段目录
5. 查看 AI 候选结果
6. 进行人工审核

第一阶段不建议优先做：

1. 全量图谱编辑器
2. 任意拖拽建边
3. 复杂可视化公式编辑
4. 全功能配置管理后台

### 20.3 第一阶段核心页面

#### Graph Overview

用于查看整张图谱。

建议展示：

1. 节点列表
2. 边列表
3. 节点之间的方向关系
4. 关系类型
5. 孤立节点提示
6. 高置信度 / 待确认关系标记

#### Node Detail

用于查看单个节点的详细信息。

建议展示：

1. 节点名
2. 对应表名
3. 实体键
4. 时间键
5. 粒度
6. 字段列表
7. 节点类型
8. 备注说明

#### Edge Detail

用于查看单条边的详细信息。

建议展示：

1. `from`
2. `to`
3. `relation_type`
4. `join_keys`
5. `time_binding`
6. 置信度
7. 规则推断备注
8. 是否已人工确认

#### Field Catalog

用于查看标准字段目录。

建议展示：

1. `standard_field`
2. 来源节点
3. 来源字段
4. `resolver_type`
5. 依赖字段
6. 公式
7. 适用粒度
8. 备注

### 20.4 最重要的前端能力

这个前端最值得做的能力不是“编辑”，而是：

**AI 候选结果审核**

建议支持：

1. 查看规则推断结果
2. 查看 AI refine 结果
3. 比较候选图谱和当前已确认配置
4. 标记“接受 / 待修改 / 拒绝”
5. 记录审核备注

### 20.5 第二阶段再考虑的能力

等后端协议更稳定后，再考虑增加：

1. 新增节点
2. 新增边
3. 修改 `time_binding`
4. 修改字段映射
5. 修改派生公式
6. 一键导出配置

### 20.6 与当前系统的关系

前端不是替代 CLI，也不是替代配置文件。

更合理的分工是：

1. CLI 负责扫描、推断、导出
2. 前端负责查看、审核、确认
3. 配置文件仍然是最终落盘格式

### 20.7 一句话定位

前端第一阶段应定位为：

**面向图谱与字段目录的可视化查看和审核工作台。**
